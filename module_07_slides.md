# Why Documentation Is a CI Requirement

<div class="grid grid-cols-2 gap-8 mt-4">
<div>

**The real cost of missing docs**

Three months after building a model:

> *"What does `fct_prescription.dosage_amount` represent? Is it milligrams? Is it a monetary amount? Can it be NULL?"*

If the column isn't documented, the answer requires:
- Reading the SQL
- Tracing back through staging to Bronze
- Hoping the source system has docs

That's 30 minutes of investigation that a two-sentence description would have prevented.

</div>
<div>

**What CI enforces for Silver and Gold**

<div class="space-y-2 mt-2">
  <div class="bg-red-50 border border-red-200 rounded-lg p-3 text-sm text-red-700">
    ❌ No model description → PR rejected
  </div>
  <div class="bg-red-50 border border-red-200 rounded-lg p-3 text-sm text-red-700">
    ❌ No grain statement → PR rejected
  </div>
  <div class="bg-red-50 border border-red-200 rounded-lg p-3 text-sm text-red-700">
    ❌ Undocumented columns → PR rejected
  </div>
  <div class="bg-emerald-50 border border-emerald-200 rounded-lg p-3 text-sm text-emerald-700">
    ✅ Bronze and Staging: optional (but encouraged)
  </div>
</div>

</div>
</div>

# Writing Model and Column Descriptions

```yaml {all|3-8|10-15|17-23|all}
version: 2

models:
  - name: fct_prescription
    description: >
      Grain: one prescription event per patient per doctor per prescription_date.
      Source: HubSpot deal records filtered to prescription pipeline stages.
      Refreshed nightly via Airflow. Excludes cancelled prescriptions.

    columns:
      - name: prescription_key
        description: >
          Surrogate primary key. MD5 hash of (patient_key, doctor_key, prescription_date).
          Integer. Not a UUID.
        tests: [unique, not_null]

      - name: patient_key
        description: "FK to dim_patient.patient_key. Surrogate key generated by our pipeline."
        tests:
          - not_null
          - relationships:
              to: ref('dim_patient')
              field: patient_key
```

<!--
Use line highlights: first show the model-level description block, then the prescription_key column, then the patient_key column with FK test.

Note that descriptions and tests live in the same schema.yml — they are not separate concerns. When you're writing tests, write descriptions. When you're writing descriptions, add tests. They belong together.

The grain statement format is mandatory: "Grain: one {entity} per {dimension1} per {dimension2}." If you can't write the grain statement, the model's design is unclear and needs a design review before documentation.
-->

# The Grain Statement

<div class="mt-4 bg-white border border-slate-200 rounded-xl p-6">
  <div class="font-semibold text-slate-700 mb-3">Mandatory format for Silver and Gold models</div>

```
Grain: one {entity} per {dimension1} per {dimension2} per {time dimension}.
```

</div>

<div class="mt-4 space-y-3">

<div class="bg-emerald-50 border border-emerald-200 rounded-lg p-4 text-sm">
  <div class="font-mono text-emerald-600 text-xs mb-1">✅ Good grain statement</div>
  <div>"Grain: one prescription event per patient per doctor per prescription_date."</div>
</div>

<div class="bg-red-50 border border-red-200 rounded-lg p-4 text-sm">
  <div class="font-mono text-red-500 text-xs mb-1">❌ Too vague — CI will reject</div>
  <div>"Grain: prescriptions." — What does one row represent exactly?</div>
</div>

<div class="bg-amber-50 border border-amber-200 rounded-lg p-4 text-sm">
  <div class="font-mono text-amber-600 text-xs mb-1">⚠️ If you can't write the grain, the design is unclear</div>
  <div>Stop. Review the model structure before writing documentation. A grain statement that's hard to write is a design smell.</div>
</div>

</div>

<!--
The grain statement is the most important sentence in any data model. It defines the semantic meaning of a row.

Without it, someone querying the model can't know if they should aggregate, deduplicate, or filter. With it, they can write correct SQL immediately.

The "if you can't write it, the design is unclear" point is important. A grain statement that takes three attempts is telling you something about the model — it might have mixed granularity, implicit filters, or SCD2 complications that should be made explicit.

Ask: "Write a grain statement for fct_prescription." Someone should be able to answer: "Grain: one prescription event per patient per doctor per prescription_date."
-->

# Doc Blocks — For Longer Descriptions

<div class="mt-4 grid grid-cols-2 gap-4">
<div>

**`models/silver/_silver_docs.md`**
```markdown
{% docs fct_prescription %}
One prescription per patient per doctor per date.
Source: HubSpot deals in prescription pipeline stages.
Excludes cancelled prescriptions (status != closed_won).
Refreshed nightly after Bronze load completes.
{% enddocs %}
```

</div>
<div>

**`schema.yml`**
```yaml
models:
  - name: fct_prescription
    description: '{{ doc("fct_prescription") }}'
    columns:
      - name: prescription_key
        description: "Surrogate PK. MD5 hash."
```

</div>
</div>

<div class="mt-4 bg-slate-50 border border-slate-200 rounded-lg p-3 text-sm text-slate-600">
  <strong>At Bloomwell:</strong> Short inline descriptions are the standard. Use doc blocks only when a model's business context genuinely needs more than one or two lines — typically complex Silver facts or models with non-obvious exclusion logic.
</div>

<!--
Keep this to 3 minutes. Most of the time at Bloomwell, inline descriptions are fine and preferred.

The doc block pattern is worth knowing because:
1. It exists in the Bloomwell codebase for a few complex models
2. Trainees will encounter it and wonder what {% docs %} means
3. Multi-line YAML descriptions with > syntax get unwieldy for anything beyond 2-3 sentences

The syntax trip-up: the reference in schema.yml uses single quotes around {{ doc("block_name") }} — required because the double quotes inside would break YAML.

Don't let this become a discussion of when to use inline vs. doc blocks. The rule is simple: one or two sentences → inline. More → doc block.
-->

# `persist_docs` — Docs Flow into Snowflake

<div class="grid grid-cols-2 gap-8 mt-4">
<div>

**Already configured in `dbt_project.yml`**

```yaml
models:
  bloomwell:
    silver:
      +persist_docs:
        relation: true   # model description → table comment
        columns: true    # column descriptions → column comments
```

**What dbt runs after every Silver/Gold build:**

```sql
COMMENT ON TABLE BLOOMWELL.SILVER.fct_prescription
IS 'Grain: one prescription event per...';

COMMENT ON COLUMN BLOOMWELL.SILVER.fct_prescription.prescription_key
IS 'Surrogate primary key. MD5 hash of...';
```

# Exercise: Document `dim_pipeline` (20 min)

**The model exists with no documentation. Write a complete `schema.yml` entry.**

<div class="mt-4">

| Column | Type | Description |
|---|---|---|
| `pipeline_key` | Integer | Surrogate PK |
| `hubspot_pipeline_id` | String | Business key from HubSpot |
| `pipeline_name` | String | Human-readable name |
| `is_active` | Boolean | Whether pipeline is currently in use |
| `dbt_valid_from` | Timestamp | SCD2 validity start |
| `dbt_valid_to` | Timestamp | SCD2 validity end (NULL = current record) |
| `is_current` | Boolean | True if active version of this record |

</div>

<div class="mt-4 bg-amber-50 border border-amber-200 rounded-lg p-3 text-sm text-amber-700">
  <strong>Hint on grain:</strong> This is SCD2 — one row per pipeline per validity period. The grain must reflect that.
</div>

<!--
This exercise tests whether they can apply the grain statement format to a non-trivial case (SCD2).

Expected grain statement: "Grain: one pipeline record per validity period. SCD2 tracked — dbt_valid_to IS NULL identifies the current active record."

Required tests:
- pipeline_key: unique + not_null (surrogate PK)
- hubspot_pipeline_id: not_null (business key — not necessarily unique due to SCD2 history)
- is_current: accepted_values [true, false]

After writing: run dbt docs generate → dbt docs serve → find dim_pipeline in the docs. Verify the description and grain appear correctly.
